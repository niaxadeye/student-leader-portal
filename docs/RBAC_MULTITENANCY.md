# Корректировки: мульти-арендность, роли и персональные уведомления

_Спецификация готова к реализации. Расширяет SITE.md §5–6, §15, §21. Все уточнения
(модель + 8 открытых вопросов O1–O8) утверждены заказчиком 2026-07-14. См. §7._

## 0. Краткая суть изменений

Система превращается из «одной инсталляции с одним владельцем» в **мульти-арендную
платформу**, где работают несколько независимых организаторов. Вводится над-роль
**MEGA_ADMIN**, а **SUPER_ADMIN** становится «организатором» — владельцем своих
данных, которые он не делит с другими организаторами.

Четыре опорных решения (утверждены заказчиком):

1. **Изоляция данных — по владельцу (`created_by`)**, а НЕ по организации.
   Два суперадмина одной организации НЕ видят данные друг друга. Организация —
   это ярлык-атрибут (для отображения и, возможно, группировки), не граница доступа.
2. **MEGA_ADMIN** видит абсолютно всё и создаёт организаторов (SUPER_ADMIN).
   SUPER_ADMIN не может создавать других SUPER_ADMIN.
3. **Доступ админа к конкурсу — per-contest, с уровнем EDIT или VIEW.** Один админ
   может иметь EDIT на конкурсе A и VIEW на конкурсе B.
4. **Уведомления — персональная привязка Telegram** у каждого админа/суперадмина
   плюс подписка на конкретные конкурсы. Общий чат из env уходит (или остаётся
   лишь как fallback).

Дополнительно утверждено:

- Организация есть **только у ADMIN/SUPER_ADMIN**, не у конкурсантов (у конкурсанта
  остаётся своё поле «организация/вуз» как сейчас — это разные вещи, см. §2.3).
- Организация **наследуется от создателя**: MEGA_ADMIN задаёт её при создании
  SUPER_ADMIN; созданные суперадмином админы наследуют его организацию.
- **Конкурсантов и админов создаёт только SUPER_ADMIN.** EDIT-админ НЕ управляет
  участниками вообще (даже привязать существующего не может) — только контент
  конкурса (испытания, формы, просмотр/проверка работ).
- SUPER_ADMIN — **полный владелец своих конкурсов** (неявный EDIT на все, что создал;
  отдельное назначение себе не нужно).
- **Существующие данные можно сбросить** — миграция не обязана переносить старое.

## 1. Роли и матрица прав

### 1.1. Иерархия ролей (обновлённая)

| Роль | Уровень | Кто создаёт | Видимость | Ключевые права |
|------|---------|-------------|-----------|----------------|
| `MEGA_ADMIN` | платформа | env-bootstrap (первый) / другой MEGA_ADMIN | **всё** без ограничений | Всё, что могут остальные + управление организаторами и организациями. Кросс-арендный доступ. |
| `SUPER_ADMIN` (организатор) | арендатор | только MEGA_ADMIN | только созданное **им самим** (цепочка `created_by`) | Создаёт конкурсы (владелец), заводит админов и конкурсантов, назначает доступ админам, полный EDIT на свои конкурсы. |
| `ADMIN` | конкурс | SUPER_ADMIN | только конкурсы, где есть назначение (+ конкурсанты этих конкурсов) | Per-contest EDIT или VIEW. НЕ управляет составом участников. НЕ создаёт пользователей. |
| `CONTESTANT` | участник | SUPER_ADMIN | только свои конкурсы/работы | Как сейчас (черновики, отправка, файлы). |

Приоритет для «основной роли» в JWT (login): `MEGA_ADMIN` > `SUPER_ADMIN` > `ADMIN` > `CONTESTANT`.

### 1.2. Уровни доступа админа к конкурсу

Новое понятие **access level** на назначении «ADMIN + конкурс»:

- **EDIT** — редактирует испытания, формы, схему; видит и проверяет работы; НЕ трогает состав участников (это только SUPER_ADMIN).
- **VIEW** — только чтение: видит конкурс, испытания, работы, но не может ничего менять.

SUPER_ADMIN на своих конкурсах = неявный EDIT (без строки назначения — доступ по владению).

### 1.3. Матрица действий (сводно)

| Действие | MEGA | SUPER (владелец) | ADMIN EDIT | ADMIN VIEW | CONTESTANT |
|----------|:----:|:----------------:|:----------:|:----------:|:----------:|
| Создать SUPER_ADMIN | ✅ | ❌ | ❌ | ❌ | ❌ |
| Создать ADMIN | ✅ | ✅ (свой) | ❌ | ❌ | ❌ |
| Создать конкурсанта | ✅ | ✅ (свой) | ❌ | ❌ | ❌ |
| Создать конкурс | ✅ | ✅ (владелец) | ❌ | ❌ | ❌ |
| Редактировать конкурс/испытания/форму | ✅ | ✅ (свои) | ✅ | ❌ | ❌ |
| Назначить админу доступ к конкурсу | ✅ | ✅ (свои) | ❌ | ❌ | ❌ |
| Управлять участниками конкурса | ✅ | ✅ (свои) | ❌ | ❌ | ❌ |
| Смотреть/проверять работы | ✅ | ✅ (свои) | ✅ | ✅ | ❌ (свои) |
| Настроить свои уведомления | ✅ | ✅ | ✅ | ✅ | ❌ |
| Видеть всё в системе | ✅ | ❌ | ❌ | ❌ | ❌ |

## 2. Модель данных (изменения схемы)

### 2.1. Роль MEGA_ADMIN + уровень доступа

Миграция `0010_multitenancy.up.sql`:

```sql
-- Новая над-роль.
INSERT INTO roles (code, name) VALUES ('MEGA_ADMIN', 'Мегаадминистратор');

-- Уровень доступа на назначении ADMIN→конкурс (EDIT|VIEW).
-- Для не-ADMIN и GLOBAL-строк остаётся NULL.
ALTER TABLE user_roles ADD COLUMN access_level VARCHAR(8) NULL;
-- допустимые значения проверяются в приложении: 'EDIT' | 'VIEW'
```

### 2.2. Владение (`created_by`) — основа изоляции

```sql
-- Кто создал пользователя (организатор для админов/конкурсантов; MEGA для организаторов).
ALTER TABLE users ADD COLUMN created_by UUID NULL REFERENCES users(id);
-- Кто владелец конкурса (SUPER_ADMIN-организатор). NOT NULL после сброса данных.
ALTER TABLE contests ADD COLUMN owner_user_id UUID NULL REFERENCES users(id);
CREATE INDEX idx_users_created_by ON users(created_by);
CREATE INDEX idx_contests_owner ON contests(owner_user_id);
```

**Правило видимости (реализуется в репозиториях, НЕ в middleware):**
- MEGA_ADMIN — без фильтра.
- SUPER_ADMIN — `WHERE owner_user_id = :me` (конкурсы) / `WHERE created_by = :me` (пользователи).
- ADMIN — конкурсы через `user_roles` (scope=CONTEST); пользователи-конкурсанты этих конкурсов.

### 2.3. Организация

Заказчик: организация есть только у админов/суперадминов и наследуется от создателя.
Текущее свободнотекстовое `users.organization` уже используется для **конкурсанта**
(его вуз/организация, показывается в таблице работ, SITE.md §7.6). Это другая сущность.

Решение: не смешивать. Ввести отдельное поле для организации-арендатора:

```sql
-- Организация-арендатор (ярлык), наследуется по цепочке создателя. Только у ADMIN/SUPER_ADMIN.
ALTER TABLE users ADD COLUMN org_name TEXT NULL;
```

**Решение (O1): организация — денормализованный ярлык `org_name`**, отдельной таблицы
`organizations` НЕ вводим. MEGA_ADMIN задаёт `org_name` при создании SUPER_ADMIN;
созданные им ADMIN наследуют это значение. Изоляция всё равно по `created_by`, поэтому
справочник для прав не нужен. Если позже понадобится брендирование (логотип/реквизиты) —
выделим таблицу отдельной миграцией, `org_name` станет FK.

### 2.4. Персональные уведомления (Telegram + подписки)

```sql
-- Привязка Telegram к пользователю (админ/суперадмин/мега). Chat_id личного диалога с ботом.
CREATE TABLE user_telegram (
    user_id     UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    chat_id     TEXT NULL,                 -- заполняется после /start с токеном
    link_token  TEXT NULL,                 -- одноразовый токен привязки (deep-link)
    linked_at   TIMESTAMPTZ NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- O8: chat_id глобально уникален (один Telegram = один аккаунт системы).
CREATE UNIQUE INDEX uq_user_telegram_chat ON user_telegram(chat_id) WHERE chat_id IS NOT NULL;

-- На какие конкурсы пользователь хочет получать уведомления.
-- O2: подписка по умолчанию ВКЛЮЧЕНА (opt-out). Отсутствие строки = подписан.
-- Строка появляется, только когда пользователь ЯВНО отключил конкурс (enabled=false).
CREATE TABLE notification_subscriptions (
    user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    contest_id UUID NOT NULL REFERENCES contests(id) ON DELETE CASCADE,
    enabled    BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, contest_id)
);
```

**Семантика подписки (O2, opt-out):** получатель подписан на конкурс, если у него есть
доступ к конкурсу И нет строки `notification_subscriptions` с `enabled=false` по этой паре.
То есть по умолчанию все, кому дали доступ, получают уведомления, пока сами не отключат.

### 2.5. Доставки outbox (O3, строка на получателя)

`outbox_events` остаётся «фактом события» (одна строка на submit). Доставка каждому
получателю — отдельная строка со своим статусом/ретраями, чтобы сбой одного адресата
не блокировал остальных:

```sql
CREATE TABLE outbox_deliveries (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_id     UUID NOT NULL REFERENCES outbox_events(id) ON DELETE CASCADE,
    user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    chat_id      TEXT NOT NULL,                       -- снимок на момент фан-аута
    status       VARCHAR(16) NOT NULL DEFAULT 'PENDING', -- PENDING|SENT|DEAD
    attempts     INT NOT NULL DEFAULT 0,
    available_at TIMESTAMPTZ NOT NULL DEFAULT now(),  -- backoff
    locked_at    TIMESTAMPTZ NULL,
    locked_by    TEXT NULL,
    last_error   TEXT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (event_id, user_id)
);
CREATE INDEX idx_deliveries_claim ON outbox_deliveries(status, available_at);
```

Событие `outbox_events` считается обработанным (fan-out выполнен), когда по нему созданы
все строки `outbox_deliveries`. Диспетчер дальше поллит именно `outbox_deliveries`.

## 3. Изменения в бэкенде

### 3.1. JWT и принципал (`platform/security/jwt.go`, `middleware`)

- Роль `MEGA_ADMIN` в приоритете основной роли (login `primaryRole`, `auth/service.go:85`).
- `middleware/rbac.go:24` — сейчас хардкод `p.Role == "SUPER_ADMIN"` даёт полный проход.
  Добавить: `MEGA_ADMIN` проходит везде; `SUPER_ADMIN` больше НЕ «проходит везде» на
  чужих данных — его всевластие ограничивается фильтром владения в репозиториях.
- `Principal` можно дополнить `OrgName` (для UI), но права считаются по владению, не по орг.

### 3.2. Проверка доступа к конкурсу (`contests/repo.go:HasContestAccess`)

Переписать под три уровня:

```
accessLevel(user, contest) →
  MEGA_ADMIN                         → EDIT   (O5: полный доступ ко всему, вкл. правку чужого)
  contest.owner_user_id == user      → EDIT   (владелец-организатор)
  user_roles(ADMIN, contest).level   → EDIT | VIEW
  иначе                              → нет доступа
```

**O5: MEGA_ADMIN — полное редактирование всего**, а не только просмотр. Он проходит
как EDIT на любой конкурс/работу любого организатора. Все действия MEGA пишутся в аудит
(`audit_log`), т.к. это кросс-арендное вмешательство.

Все мутационные хендлеры (создание/правка испытаний, форм, схемы) требуют EDIT.
Чтение (просмотр работ, испытаний) — EDIT или VIEW. Ввести хелпер
`RequireContestEdit` / `RequireContestView` вместо булева `HasContestAccess`.

### 3.3. Создание пользователей (`useradmin/`)

- `ValidRoles` (`useradmin/types.go:6`) — добавить `MEGA_ADMIN`.
- **Правила создания (новый guard в сервисе, не только в роутере):**
  - `MEGA_ADMIN` может создать любую роль.
  - `SUPER_ADMIN` может создать только `ADMIN` и `CONTESTANT`; НЕ `SUPER_ADMIN`/`MEGA_ADMIN`.
  - При создании проставлять `created_by = actor.UserID` и `org_name` (наследование от создателя для ADMIN).
- **Список пользователей** (`repo.go`) фильтруется:
  - MEGA — все; SUPER — `WHERE created_by = :me`; ADMIN — нет доступа к реестру.
- Роутер `router.go:104` (`/users` под `RequireRole("SUPER_ADMIN")`) → пустить и `MEGA_ADMIN`.

### 3.4. Назначение доступа админам (`useradmin/handlers_roles.go`)

- `AssignRoleInput` — добавить `AccessLevel` (`EDIT`|`VIEW`) для назначений ADMIN+CONTEST.
- Guard: назначать доступ к конкурсу может только владелец конкурса (SUPER_ADMIN) или MEGA_ADMIN.
- Валидация: `access_level` обязателен при `role=ADMIN, scope_type=CONTEST`; иначе NULL.

### 3.5. Создание конкурса (`contests/service.go:57`)

- Проставлять `owner_user_id = actor.UserID` при создании.
- `ListForPrincipal` (`contests/repo.go:33`) переписать:
  - MEGA — все конкурсы;
  - SUPER — `WHERE owner_user_id = :me`;
  - ADMIN — конкурсы из `user_roles` (как сейчас), плюс отдавать в JSON их `access_level`.

### 3.6. Участники конкурса (`contests/service_participants.go`)

- Управление участниками (add/remove/import) — только владелец (SUPER_ADMIN) или MEGA.
- EDIT-админ получает 403 на эти эндпоинты (сейчас доступ по `HasContestAccess` — ужесточить).

### 3.7. Outbox-диспетчер → персональные адресаты (`modules/outbox/`)

Сейчас `dispatcher.go` шлёт в один `DefaultChat` из env. Двухфазная схема (O3):

**Фаза 1 — fan-out.** По PENDING `outbox_events` определить **список получателей**:
пользователи с доступом к конкурсу (владелец + назначенные админы), у которых привязан
Telegram (`user_telegram.chat_id IS NOT NULL`) И подписка не отключена (нет строки
`notification_subscriptions` с `enabled=false`). Для каждого — вставить строку
`outbox_deliveries` (идемпотентно по `UNIQUE(event_id, user_id)`). Событие помечается
обработанным (fan-out завершён).

**Фаза 2 — доставка.** Поллинг `outbox_deliveries` (`FOR UPDATE SKIP LOCKED`),
как сейчас в диспетчере: exponential backoff, DEAD после лимита — но **по каждому
получателю независимо**. Сбой одного `chat_id` не влияет на остальных.

- Env `TELEGRAM_DEFAULT_CHAT_ID` — оставить как опциональный fallback/лог-канал.
- Текст сообщения (`format.go`) не меняется; резолвер полей — тот же.

### 3.8. Новые эндпоинты

```
POST   /api/v1/me/telegram/link         → выдать deep-link токен (t.me/bot?start=<token>)
DELETE /api/v1/me/telegram              → отвязать
GET    /api/v1/me/notifications         → список конкурсов + флаг подписки
PUT    /api/v1/me/notifications/:cid    → вкл/выкл подписку на конкурс
POST   /api/v1/telegram/webhook/:secret → приём /start от бота (привязка chat_id по токену)
```

**Telegram-бот — webhook (O7).** Приём апдейтов через публичный HTTPS-эндпоинт (nginx
уже есть). При старте регистрируем `setWebhook` на `https://eazytech.ru/api/v1/telegram/webhook/<secret>`
(секрет в пути — защита от подделки, плюс проверка `X-Telegram-Bot-Api-Secret-Token`).
Флоу привязки: пользователь жмёт «Привязать Telegram» → бэкенд генерит `link_token` →
даём ссылку `t.me/<bot>?start=<link_token>` → пользователь в Telegram жмёт Start →
Telegram шлёт `/start <link_token>` на webhook → бэкенд находит юзера по токену,
сохраняет `chat_id`, ставит `linked_at`, гасит токен. Long-polling не используем.

### 3.9. Заморозка организатора (O6)

Блокировка/«удаление» SUPER_ADMIN — это НЕ физическое удаление, а `users.status = 'BLOCKED'`
(механизм уже есть). Данные организатора (конкурсы, конкурсанты, админы, работы) остаются
в БД нетронутыми:

- Заблокированный организатор не может логиниться (проверка статуса в auth уже есть).
- Его конкурсы продолжают существовать; при необходимости MEGA_ADMIN видит и правит их
  (полный доступ по O5) или разблокирует организатора обратно.
- Каскадного удаления НЕТ. Обратимо. Только MEGA_ADMIN может блокировать/разблокировать
  SUPER_ADMIN.
- Открыто на будущее: нужен ли автоматический «морозильник» для конкурсантов
  заблокированного организатора (запрет входа). Пока — по требованию, не в MVP.

Заказчик разрешил **сброс данных**. План:

- `smoke_outbox.sh` — проверить fan-out в `outbox_deliveries` и доставку персонально
  подписанному пользователю (а не в общий чат); отключение подписки → нет строки доставки.
- Новый `smoke_multitenancy.sh` — сквозной: MEGA создаёт двух организаторов из разных орг,
  каждый создаёт конкурс + админа (EDIT и VIEW), проверка невидимости чужого, подписки.

## 7. Утверждённые решения (все вопросы закрыты 2026-07-14)

- **O1. Организация — ярлык.** Денормализованный `users.org_name` (текст), наследуется
  от создателя. Отдельной таблицы `organizations` пока нет (см. §2.3).
- **O2. Подписка по умолчанию ВКЛЮЧЕНА (opt-out).** Кому дали доступ к конкурсу —
  получает уведомления, пока сам не отключит. Отсутствие строки = подписан (см. §2.4).
- **O3. Строка на получателя.** Таблица `outbox_deliveries` (event_id, user_id, chat_id,
  status), независимые ретраи/DEAD по каждому адресату. Двухфазный диспетчер (см. §2.5, §3.7).
- **O4. Только отправка/обновление работы** (`submission.submitted`/`resubmitted`) — как
  сейчас. Напоминания о дедлайне, «новый участник», «ошибка файла» — отложены (см. §9).
- **O5. MEGA_ADMIN — полное редактирование всего**, не только просмотр. Действия MEGA
  пишутся в аудит как кросс-арендные (см. §3.2).
- **O6. Заморозка.** Блокировка/удаление SUPER_ADMIN = смена статуса на BLOCKED; его
  конкурсы/конкурсанты/админы остаются в БД нетронутыми, доступ закрыт, но обратимо
  (см. §3.9).
- **O7. Webhook.** Бот принимает `/start` через публичный HTTPS-эндпоинт с секретом в
  пути + `X-Telegram-Bot-Api-Secret-Token`. Long-polling не используем (см. §3.8).
- **O8. chat_id глобально уникален.** Один Telegram-аккаунт = один аккаунт системы
  (частичный UNIQUE-индекс, см. §2.4).

## 8. Порядок реализации (предложение)

1. Миграция `0010` (роль MEGA_ADMIN, `created_by`/`owner_user_id`, `access_level`,
   `org_name`, `user_telegram`, `notification_subscriptions`, `outbox_deliveries`) +
   сброс данных + bootstrap MEGA_ADMIN из env.
2. Изоляция чтения по владению (contests/users repos) + guard создания ролей
   (SUPER создаёт только ADMIN/CONTESTANT со своим `created_by`/`org_name`).
3. Уровни EDIT/VIEW: `accessLevel()`, `RequireContestEdit/View`, ужесточение мутаций;
   участниками управляет только владелец/MEGA.
4. Фронт: гарды ролей (вкл. MEGA), VIEW-режим (дизейбл правок), экраны «Организаторы»
   и реестр по владению, выбор EDIT/VIEW при назначении.
5. Персональные уведомления: таблицы, эндпоинты `/me/...`, Telegram-бот привязка (webhook).
6. Outbox → двухфазный (fan-out в `outbox_deliveries` + доставка per-recipient, O3).
7. Смоук-тесты (`smoke_multitenancy.sh` + правки существующих) и регресс.
